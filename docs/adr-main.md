# ADR: Архитектура универсальной системы самообновления (Launcher-Core)

## 1. Статус

Принято (Accepted) — Версия 5.1 (итоговая, расширенная)

## 2. Контекст

Создание универсального набора инструментов для сборки **портативных Windows-приложений** с механизмом прозрачного самообновления. Система должна быть полностью абстрагирована от конкретного ПО, обеспечивать безопасность данных и позволять собирать брендированные лаунчеры из единого исходного кода. Решение ориентировано только на ОС Windows; для macOS и Linux будут использоваться иные механизмы.

## 3. Компоненты системы

- **Launcher Source**: Репозиторий с исходным кодом на Rust. Содержит шаблон лаунчера и **утилиту `updater-tool`** для работы с манифестами и ключами. Не содержит жёстких имён, URL или ключей.
- **App Project**: Репозиторий конкретного приложения, выступающий контекстом сборки.
- **Updater-Tool**: Входит в состав Launcher Source. CLI-утилита, которая:
  - генерирует пару ключей Ed25519 (приватный/публичный) и сохраняет их в файлы;
  - создаёт подписанный манифест `update.json` для заданного ZIP-архива (вычисляет SHA256, формирует подпись);
  - может использоваться как в CI/CD, так и локально для начальной настройки.

## 4. Архитектурное решение

### 4.1. Локальное окружение (у пользователя)

```
/Root
├── [APP_NAME].exe          # Launcher (точка входа)
├── updater.log             # Лог с временными метками и уровнями важности (с ротацией)
└── /bin                    # Изолированная директория для ядра и метаданных
    ├── [APP_NAME_CORE].exe # Исполняемое ядро (Core)
    └── manifest.json       # Локальный файл с текущей версией и хешем ядра
```

**Локальный `manifest.json`** содержит:

```json
{
  "version": "1.2.3",
  "sha256": "хеш_текущего_ядра"
}
```

### 4.2. Жизненный цикл процесса (Lifecycle)

1. **Инициализация и очистка**
   - Открывается (создаётся) `updater.log`, записывается начало сессии.
   - Выполняется ротация лога: при превышении размера 1 МБ текущий файл переименовывается в `updater.log.old`, создаётся новый.
   - Удаляются временные артефакты предыдущих сеансов (`*.tmp`, `*.old`, папка `bin.old`), если они старше 5 минут. При успешном завершении предыдущего обновления временные файлы удаляются немедленно.

2. **Проверка целостности и версии ядра**
   - Если файл `bin/[APP_NAME_CORE].exe` отсутствует, локальная версия считается равной `0.0.0` (триггер полной загрузки).
   - Если файл присутствует, вычисляется его SHA256 (занимает ~0.1–0.3 с для 50–100 МБ) и сравнивается со значением из локального `manifest.json`. При несовпадении:
     - в лог записывается предупреждение `WARN` с указанием несоответствия хеша;
     - локальная версия приравнивается к `0.0.0`, что инициирует загрузку обновления.

3. **Блокировка активного процесса**
   - Проверяется, не запущено ли ядро. Рекомендуемый механизм: ядро при старте создаёт именованный мьютекс `Global\APP_NAME_CORE_MUTEX`. Лаунчер пытается открыть этот мьютекс; если успешно — ядро активно.
   - Если ядро запущено, обновление не выполняется, лаунчер сразу переходит к запуску ядра (п. 15) или завершается с сообщением о необходимости закрыть приложение (если запуск не предполагается).

4. **Запрос удалённого манифеста**
   - По вшитому `MANIFEST_URL` (обязательно HTTPS) скачивается `update.json` с применением таймаутов (раздел 7). Лаунчер следует HTTP-редиректам (до 5 переходов).

5. **Верификация подписи манифеста (Ed25519)**
   - Подпись проверяется вшитым публичным ключом. Подписывается каноническая строка, составленная из полей `version`, `url`, `sha256` (конкатенация без разделителей).
   - При недействительной подписи процесс прерывается с записью ошибки в лог.

6. **Сравнение версий**
   - Версии сравниваются семантически (например, через крейт `semver`). Если версия из манифеста **строго больше** локальной (или локальная версия `0.0.0`) — инициируется обновление.
   - В противном случае (например, версия не больше, но ядро существует) — переход к п. 15 (запуск ядра).

7. **Обработка ошибок сети**
   - Если на шаге 4 произошла ошибка соединения (таймаут, недоступность хоста), но локальное ядро существует и его хеш совпадает, лаунчер НЕ прерывает работу, а сразу переходит к запуску ядра (п. 15), записав в лог предупреждение `WARN: Network unavailable, using local version`. Это обеспечивает работу в офлайн-режиме.

8. **Загрузка пакета обновления**
   - По URL из манифеста скачивается `Package.zip` во временный файл `Package.tmp` с контролем активности потока (таймауты из раздела 7).
   - При ошибке загрузки (сетевая, таймаут) временный файл удаляется, процесс завершается с ошибкой (если ядро отсутствует или повреждено, лаунчер не может продолжить).

9. **Проверка целостности архива**
   - Вычисляется SHA256 от `Package.tmp` и сравнивается со значением из манифеста. При несовпадении временный файл удаляется, процесс завершается с ошибкой.

10. **Подготовка временной структуры**
    - Содержимое `Package.tmp` распаковывается в новую папку `bin.new`, расположенную рядом с существующей `bin`.
    - В архиве ожидаются: новый лаунчер `[APP_NAME].exe` и папка `bin/` с актуальным ядром. Все файлы из `bin/` копируются в `bin.new` с сохранением относительных путей.

11. **Самообновление лаунчера (экспериментальная функция)**
    - Вычисляется SHA256 текущего лаунчера и сравнивается с SHA256 лаунчера из архива. При различии запускается механизм самозамены.
    - **Реализация (требует реального тестирования):**
      1. Новый лаунчер сохраняется как `[APP_NAME].new`.
      2. Текущий процесс переименовывается в `[APP_NAME].old`.
      3. Файл `[APP_NAME].new` переименовывается в `[APP_NAME].exe`.
      4. Запускается новый процесс (с теми же аргументами командной строки).
      5. При старте новый процесс проверяет наличие `[APP_NAME].old` и удаляет его.
    - **Отказоустойчивость:** Если новый процесс не запустился (например, из-за повреждения), при следующем запуске (пользователем) лаунчер должен обнаружить наличие `.old` файла и восстановить его (переименовать обратно в `.exe`), записав предупреждение в лог.

12. **Атомарная замена папки ядра**
    - Существующая папка `bin` переименовывается в `bin.old`.
    - Временная папка `bin.new` переименовывается в `bin`.
    - Если обе операции переименования успешны, `bin.old` удаляется. В противном случае выполняется откат: `bin.old` возвращается в `bin`, а `bin.new` удаляется. Обновление считается неудавшимся.

13. **Удаление временного архива**
    - После успешной замены (или при откате) файл `Package.tmp` немедленно удаляется.

14. **Обновление локального манифеста**
    - В новую папку `bin` записывается обновлённый `manifest.json` с актуальной версией и хешем нового ядра.

15. **Запуск ядра**
    - Все аргументы командной строки, переданные лаунчеру, без изменений передаются процессу `bin/[APP_NAME_CORE].exe`.
    - Ядро запускается как отдельный процесс, лаунчер завершает работу.

### 4.3. Спецификация удалённого манифеста (Update Contract)

Манифест (`update.json`) – JSON-объект:

```json
{
  "version": "1.2.3",
  "url": "https://example.com/download/Package.zip",
  "sha256": "e3b0c44298fc1c149afbf4c8996fb92427ae41e4649b934ca495991b7852b855",
  "signature": "ed25519_base64_подпись"
}
```

Подпись формируется утилитой `updater-tool` путём подписывания канонической строки `<version><url><sha256>` приватным ключом.

### 4.4. Параметризация при компиляции (Build-time Injection)

В код лаунчера через переменные окружения внедряются:

- `APP_NAME` – имя приложения (для имён файлов);
- `MANIFEST_URL` – постоянная ссылка на `update.json` (HTTPS);
- `PUBLIC_KEY` – публичный ключ Ed25519 в hex-формате.

В Rust доступ через `env!("VAR")`.

### 4.5. Механизм обновления (Single‑Package)

- **Формат поставки**: ZIP-архив (`Package.zip`), содержащий новый лаунчер и папку `bin/` с актуальным ядром.
- **Транзакционность**: все операции записи во временные области; финальная замена происходит только после полной проверки.
- **Атомарность на уровне папки**: замена `bin` через переименование всей директории гарантирует, что обновление либо полностью применится, либо не затронет работающую версию.

### 4.6. Проброс аргументов командной строки

Лаунчер не анализирует и не изменяет аргументы вызова – они прозрачно передаются ядру.

### 4.7. CI/CD и межрепозиторное взаимодействие

#### 4.7.1. Репозиторий Launcher Source

- Содержит код лаунчера и исходники `updater-tool`.
- При изменениях формата манифеста обновляются обе части синхронно.
- Релизы не выпускаются; код используется как зависимость в CI/CD приложений.

#### 4.7.2. Репозиторий приложения (App)

При создании релиза в CI (например, GitHub Actions) выполняются шаги:

1. **Клонирование Launcher Source** – получение актуальных исходников лаунчера и `updater-tool`.
2. **Компиляция лаунчера с инъекцией** – сборка `[APP_NAME].exe` с подстановкой `APP_NAME`, `MANIFEST_URL`, `PUBLIC_KEY`.
3. **Сборка ядра** – компиляция `[APP_NAME_CORE].exe` из репозитория приложения.
4. **Бандлинг** – создание `Package.zip` (лаунчер + папка `bin/` с ядром).
5. **Подпись манифеста с помощью `updater-tool`**:
   - Утилита компилируется в процессе CI (рекомендуется для гарантии свежести кода; при желании можно использовать предварительно собранный бинарный файл для ускорения).
   - `updater-tool sign --private-key <key> --archive Package.zip --output update.json`
6. **Публикация** – `Package.zip` и `update.json` загружаются в релиз.

### 4.8. Утилита Updater-Tool: детали работы

`updater-tool` предоставляет две основные команды:

- **`generate-keys`** – создаёт новую пару ключей Ed25519.
  - Пример: `updater-tool generate-keys --out-dir ./keys`
  - Результат:
    - `private_key.pem` – приватный ключ в формате PEM (содержит только ключ, без парольной фразы). Этот файл должен храниться в секретах CI/CD и никогда не распространяться.
    - `public_key.hex` – публичный ключ в шестнадцатеричном представлении (64 символа). Именно это значение впоследствии вшивается в лаунчер как `PUBLIC_KEY`.
  - Опционально может выводить публичный ключ в терминал для удобства копирования.

- **`sign`** – создаёт подписанный манифест для указанного ZIP-архива.
  - Пример: `updater-tool sign --private-key ./keys/private_key.pem --archive Package.zip --output update.json`
  - Вычисляет SHA256 архива, формирует каноническую строку из версии (которая извлекается из аргумента или файла), URL и хеша, подписывает её и сохраняет манифест.

Утилита написана на Rust и входит в состав репозитория Launcher Source, что гарантирует согласованность форматов между подписывающим инструментом и проверяющим кодом лаунчера.

## 5. Безопасность

- **Ed25519** – гарантия авторства манифеста. Приватный ключ хранится в секретах CI.
- **SHA256** – контроль целостности загружаемого архива и локального ядра.
- **HTTPS** – обязателен для всех URL (манифеста и скачивания), чтобы предотвратить MITM-атаки.
- **Атомарные переименования** – исключают повреждение приложения при сбоях.
- **Блокировка процессов через мьютекс** – предотвращает замену файлов во время работы ядра.

## 6. Обработка ошибок и логирование

- **Лог-файл `updater.log`** содержит временные метки и уровни (INFO, WARN, ERROR) для всех этапов.
- **Ротация лога**: при превышении 1 МБ файл переименовывается в `.old`, новый лог начинается с текущей сессии.
- **Восстановление**:
  - При отсутствии/повреждении ядра версия сбрасывается в `0.0.0` и инициируется загрузка.
  - При ошибках загрузки/распаковки временные файлы удаляются, состояние остаётся неизменным.
  - При ошибках сети, если ядро исправно, лаунчер переходит к запуску (офлайн-режим).
- **Очистка**: при старте удаляются временные артефакты старше 5 минут; после успешного обновления временные файлы удаляются немедленно.

## 7. Сетевая устойчивость (Network Resilience)

Для обеспечения отзывчивости приложения и предотвращения зависаний установлены следующие таймауты:

- **Connect Timeout**: 5 секунд на установку TCP-соединения.
- **Read Timeout**: 5 секунд на ожидание очередного блока данных после установки соединения.
- **Stall Detection**: если в течение 7 секунд не поступает новых данных, загрузка прерывается с ошибкой `STALLED_CONNECTION`.

**Обоснование значений**:

- 5 секунд на соединение – достаточно для большинства сетевых условий, при этом пользователь не испытывает длительной задержки.
- 5 секунд на чтение – позволяет пережить кратковременные флуктуации, не обрывая загрузку.
- 7 секунд для stall detection – значение намеренно превышает read timeout, чтобы отличать временные паузы (например, переадресацию) от полного зависания.

При необходимости значения могут быть переопределены на этапе компиляции через дополнительные переменные окружения (например, `CONNECT_TIMEOUT`, `READ_TIMEOUT`, `STALL_TIMEOUT`).

## 8. Зоны роста (Roadmap)

- **GUI** – минимальный интерфейс на Win32 (NWG) для отображения прогресса обновления. При обнаружении повреждённого локального ядра предлагать диалог: «Ошибка целостности. Скачать исправленную версию?» с таймаутом 5 секунд и автоматическим подтверждением «Скачать».
- **Rollback** – сохранение резервной копии `bin.old` для автоматического отката при краше новой версии (если ядро не запускается, лаунчер может восстановить предыдущую версию).
- **Докачка (resume)** – поддержка HTTP Range для возобновления прерванной загрузки больших архивов.
- **Подпись компонентов** – дополнительная проверка подписи исполняемых файлов.
- **Поддержка дополнительных файлов и миграций** – в текущей версии обновление заменяет только лаунчер и ядро. В будущем может потребоваться обновление сопутствующих данных (например, файлов конфигурации, SQLite-баз данных) или выполнение миграций схемы. Предполагаемые направления развития:
  - Расширение манифеста полем `migrations` (список скриптов или инструкций) или включение в архив дополнительных каталогов (`data/`, `migrations/`).
  - После замены ядра лаунчер (или само ядро при первом запуске) может проверять версию данных и применять необходимые миграции (например, запускать встроенные скрипты SQL).
  - Для портативных приложений важно сохранить возможность отката изменений данных, поэтому потребуется механизм резервного копирования перед миграцией.
  - Данная функциональность будет прорабатываться по мере появления реальных кейсов и включаться в новые версии ADR.
- **Поддержка нескольких каналов обновлений (stable, beta, nightly) в одном лаунчере**  
  В текущей версии для каждого канала требуется отдельная сборка лаунчера с фиксированным `MANIFEST_URL`. В будущем планируется реализовать возможность переключения канала через конфигурационный файл или аргумент командной строки без перекомпиляции.

  **Предлагаемый подход (первоначальная реализация):**  
  Единый JSON-файл, доступный по одному URL (например, `https://example.com/channels.json`), содержащий массив объектов. Каждый объект включает:
  - `channel` — имя канала (например, "stable", "beta", "nightly");
  - `version` — версия сборки для этого канала;
  - `url` — URL ZIP-архива с обновлением для данного канала;
  - `sha256` — хеш архива;
  - `signature` — подпись Ed25519, вычисленная от канонической строки `<version><url><sha256>` с использованием **единого приватного ключа** для всех каналов.

  Лаунчер при запуске:
  - Загружает этот файл по вшитому URL (или URL, полученному из конфигурации).
  - Извлекает из конфигурации или аргумента командной строки имя желаемого канала.
  - Находит в массиве объект с соответствующим `channel`.
  - Проверяет подпись этого объекта с помощью вшитого публичного ключа.
  - Далее действует по стандартному сценарию обновления (сравнение версий, загрузка и т.д.).

  **Условия и ограничения:**
  - Все каналы подписываются одним и тем же приватным ключом. Лаунчер содержит один публичный ключ.
  - Файл со списком каналов доставляется по HTTPS, что гарантирует целостность при передаче и предотвращает подмену списка на лету.
  - Добавление нового канала или изменение параметров существующего не требует перекомпиляции лаунчера — достаточно обновить `channels.json` на сервере.

  **Преимущества:**
  - Один URL для получения информации обо всех каналах.
  - Один сетевой запрос для получения всех манифестов.
  - Простота реализации и прозрачность структуры.

  **Ограничения:**
  - Если в будущем потребуется разделение ключей между каналами (например, разные каналы подписываются разными приватными ключами), этот подход потребует доработки. Возможные решения: встраивание нескольких публичных ключей в лаунчер или переход на иерархическую подпись (корневой манифест, подписывающий вложенные манифесты с их собственными ключами).
  - При компрометации приватного ключа все каналы становятся уязвимыми; потребуется перевыпуск всех манифестов и обновление лаунчеров с новым ключом.

  Выбор окончательной схемы будет уточнён при проектировании, но данный вариант принимается как базовый для прототипирования.

## 9. Рекомендации по тестированию

Для обеспечения надёжности системы перед выпуском необходимо провести тестирование в следующих сценариях:

- Обрыв соединения на разных этапах (загрузка манифеста, загрузка архива, распаковка).
- Повреждение загруженного файла (несовпадение SHA256).
- Отсутствие прав на запись в папку.
- Запуск лаунчера при активном процессе ядра.
- Сбой во время самообновления лаунчера (имитация неудачного запуска нового процесса).
- Работа в офлайн-режиме при наличии исправного ядра.
- Длительная работа с ротацией лога.
