# ADR: sign-tool — утилита для работы с ключами и подписанными манифестами

## 1. Статус

Принято (Accepted) — Версия 1.1 (обновлённая)

## 2. Контекст

В составе системы самообновления (Launcher-Core) необходима утилита, которая позволяет:

- генерировать пару ключей Ed25519 для подписи манифестов;
- создавать подписанный манифест `update_manifest.json` для готового ZIP-архива с обновлением.

Утилита должна быть удобна для использования как в CI/CD (например, GitHub Actions), так и локально разработчиками. Возникают вопросы:

- В каких форматах хранить ключи?
- Как обрабатывать ситуацию, если ключи уже существуют?
- Какие параметры необходимы для создания подписанного манифеста?
- Должна ли утилита сама загружать архив по URL или полагаться на локальный файл?
- **Как правильно организовать работу с ключами в CI/CD, чтобы приватный ключ не генерировался при каждой сборке, а безопасно использовался из защищённого хранилища?**

Данный ADR фиксирует архитектурные решения, касающиеся `sign-tool`.

## 3. Решение

### 3.1. Общее описание

`sign-tool` — это отдельная консольная утилита, входящая в состав репозитория Launcher Source. Она реализована на Rust и предоставляет две основные команды:

- `generate-keys` — создание новой пары ключей Ed25519.
- `sign` — подписание существующего ZIP-архива и генерация файла `update_manifest.json`.

### 3.2. Генерация ключей (`generate-keys`)

**Назначение:** создаёт приватный и публичный ключи для последующей подписи манифестов.

**Параметры:**

- `--out-dir <директория>` — каталог, в который будут сохранены файлы ключей. По умолчанию используется ./keys. Если указанная директория не существует, она будет создана.
  Если `--out-dir` - файл, выйти с ошибкой.
- `--force` — если ключи в целевой директории уже существуют, перезаписывать их без подтверждения. Если флаг не указан и ключи присутствуют, утилита завершается с ошибкой (чтобы случайно не уничтожить существующие ключи).

**Форматы файлов:**

- `private_key.pem` — приватный ключ в формате PEM (содержит только ключ, без парольной фразы). Пример содержимого:
  ```
  -----BEGIN PRIVATE KEY-----
  MC4CAQAwBQYDK2VwBCIEIHFQ2FojGXtQUDT+0G7I+6Q4mY6qM7f9Fw6NfJ8y4R2+
  -----END PRIVATE KEY-----
  ```
- `public_key.hex` — публичный ключ в шестнадцатеричном представлении (64 символа). Пример:
  ```
  a1b2c3d4e5f67890a1b2c3d4e5f67890a1b2c3d4e5f67890a1b2c3d4e5f67890
  ```

**Обоснование форматов:**

- PEM — стандартный формат для ключей, поддерживается многими инструментами и библиотеками.
- HEX для публичного ключа — удобно встраивать в код лаунчера как строковую константу.

**Пример использования:**

```bash
# Создать ключи в папке ./keys (если ключи уже есть — ошибка)
sign-tool generate-keys --out-dir ./keys

# Принудительно перезаписать существующие ключи
sign-tool generate-keys --out-dir ./keys --force
```

### 3.3. Подписание архива (`sign`)

**Назначение:** принимает на вход локальный ZIP-архив, публичный URL (где архив будет размещён) и версию обновления; создаёт подписанный манифест `update.json`.

**Параметры:**

- `--package <путь>` — обязательный, путь к локальному ZIP-файлу (например, `Package.zip`).
- `--url <URL>` — обязательный, публичный URL, по которому лаунчер сможет скачать этот архив. Должен использовать протокол HTTPS.
- `--version <версия>` — обязательный, строка с версией обновления. Должна соответствовать спецификации SemVer (например, 1.2.3, 2.0.0-beta.1). Утилита выполняет проверку и завершается с ошибкой, если формат невалиден.
- `--private-key <путь>` — путь к файлу приватного ключа (PEM), по умолчанию `./private_key.pem`.
- `--output <путь>` — опциональный, путь для сохранения результата. По умолчанию файл создаётся в текущей директории с именем `update_manifest.json`.
- `--force` — если выходной файл уже существует, перезаписывать его. Без этого флага при наличии существующего файла утилита завершается с ошибкой.

**Действия команды:**

1. Проверяет существование и читаемость `--package`, `--private-key`. Проверяет что `--package` - валидный ZIP архив.
2. Проверяет, что `--url` начинается с `https://`.
3. Вычисляет SHA256 хеш содержимого архива.
4. Формирует каноническую строку для подписи: конкатенация строк version, url, sha256 с разделителем `|`, именно в этом порядке. Такой подход обеспечивает детерминизм и простоту реализации.
   Например:
   ```
   "1.2.3|https://example.com/Package.zip|e3b0c44298fc1c149afbf4c8996fb92427ae41e4649b934ca495991b7852b855"
   ```
5. Подписывает полученную строку с помощью приватного ключа (алгоритм Ed25519).
6. Создаёт JSON-объект:
   ```json
   {
     "version": "1.2.3",
     "url": "https://example.com/Package.zip",
     "sha256": "вычисленный_хеш",
     "signature": "base64_подпись"
   }
   ```
7. Записывает объект в выходной файл (с отступами для читаемости).

**Пример использования (локально):**

```bash
# Создать манифест для архива Package.zip
sign-tool sign \
  --package ./release/Package.zip \
  --url "https://github.com/owner/repo/releases/download/v1.2.3/Package.zip" \
  --version "1.2.3" \
  --private-key ./keys/private_key.pem \
  --output ./release/update_manifest.json
```

### 3.4. Интеграция в CI/CD: правильное использование ключей

**Принцип:** Ключи генерируются **один раз** разработчиком локально (или администратором проекта). Приватный ключ сохраняется в защищённом хранилище CI/CD (например, в секретах GitHub Actions). В CI ключи **не генерируются заново**, а только используются для подписи.

#### 3.4.1. Первоначальная генерация ключей

Разработчик выполняет локально:

```bash
sign-tool generate-keys --out-dir ./keys
```

Полученные файлы:

- `private_key.pem` — **НИКОГДА не добавляется в репозиторий**.
- `public_key.hex` — будет встроен в лаунчер (значение передаётся при компиляции через переменную окружения `PUBLIC_KEY`).

#### 3.4.2. Сохранение приватного ключа в GitHub Secrets

- Открыть настройки репозитория → Secrets and variables → Actions.
- Создать новый секрет, например `SIGN_PRIVATE_KEY`.
- В поле значения **скопировать всё содержимое файла `private_key.pem`**, включая строки `-----BEGIN PRIVATE KEY-----` и `-----END PRIVATE KEY-----`. Секрет может содержать многострочный текст — это поддерживается.

#### 3.4.3. Использование в CI/CD (GitHub Actions)

В workflow-файле приватный ключ извлекается из секрета, записывается во временный файл, и затем передаётся утилите `sign-tool`.

**Пример корректного шага подписания:**

```yaml
- name: Sign manifest
  run: |
    # Создать временный файл с приватным ключом из секрета
    echo "${{ secrets.SIGN_PRIVATE_KEY }}" > private_key.pem
    # Запустить подписание
    sign-tool sign \
      --package ./Package.zip \
      --url "https://github.com/${{ github.repository }}/releases/download/${{ github.ref_name }}/Package.zip" \
      --version "${{ github.ref_name }}" \
      --private-key ./private_key.pem \
      --output ./update_manifest.json
    # Удалить временный файл (необязательно, т.к. раннер будет уничтожен)
    rm private_key.pem
```

**Обоснование:** Приватный ключ никогда не хранится в репозитории и не генерируется динамически, что гарантирует постоянство подписи между релизами.

### 3.5. Безопасность

- Приватный ключ **никогда не должен сохраняться в репозитории**. В CI/CD он должен храниться в секретах и передаваться через временные файлы или переменные окружения.
- Утилита не реализует шифрование ключа паролем — предполагается, что доступ к приватному ключу строго контролируется на уровне CI/CD-системы.
- Все URL должны использовать HTTPS для защиты от подмены трафика.

### 3.6. Обработка ошибок

- При отсутствии обязательных параметров выводится сообщение об ошибке и справка.
- Если указанный архив не найден или не является ZIP-файлом — ошибка.
- Если приватный ключ имеет неверный формат или повреждён — ошибка.
- При попытке перезаписать существующий выходной файл без флага `--force` — ошибка.
- Утилита завершается с ненулевым кодом возврата при любой ошибке.

### 3.7. Зоны роста (будущие возможности)

- Поддержка чтения версии из файла (например, `version.txt` внутри архива).
- Возможность указать несколько URL (для зеркал) — потребует изменения формата манифеста.
- Генерация манифеста с указанием минимальной версии лаунчера.
- Подпись отдельных файлов внутри архива (для более тонкой проверки).
- **Поддержка генерации единого файла со списком каналов**  
   Для реализации сценария с несколькими каналами в одном лаунчере необходимо расширить функциональность `sign-tool` командой, которая принимает на вход несколько ZIP-архивов (по одному на канал) с соответствующими параметрами (версия, URL, имя канала) и создаёт единый JSON-файл со списком манифестов, подписанных единым ключом.
  Должны использоваться тот же формат подписи (base64) и те же правила canonical-строки, что уже описаны.

  **Предлагаемый интерфейс команды (предварительно):**

  ```
  sign-tool generate-channels
  --channel stable:1.2.3:./stable/Package.zip:https://example.com/stable/Package.zip
  --channel beta:1.3.0-beta:./beta/Package.zip:https://example.com/beta/Package.zip
  --private-key ./private_key.pem
  --output ./channels.json
  ```

  (параметр `--channel` может повторяться; формат: `имя:версия:путь_к_архиву:url`)

  **Необходимые проверки:**
  - Все версии должны соответствовать SemVer.
  - Все SHA256 вычисляются автоматически.
  - Подпись каждого элемента формируется тем же алгоритмом, что и для одиночного манифеста (каноническая строка `version+url+sha256`).
  - Результирующий файл должен представлять собой массив объектов в формате, описанном в основном ADR.

  **Дальнейшие улучшения:**
  - Возможность указания разных ключей для разных каналов (если потребуется).
  - Валидация уникальности имён каналов.
  - Поддержка чтения параметров из YAML/TOML-файла для удобства CI/CD.

## 4. Альтернативы, которые были рассмотрены

- **Хранение приватного ключа в зашифрованном виде** — отклонено, так как в CI/CD обычно используются защищённые секреты, а дополнительное шифрование усложняет автоматизацию.
- **Генерация манифеста без локального архива (только по URL)** — отклонено, так как требует предварительной загрузки архива, что может быть неудобно в CI/CD (лишний сетевой запрос). Кроме того, вычисление хеша по удалённому файлу ненадёжно при частичной загрузке.
- **Использование единого параметра `--package` вместо раздельных `--package` и `--url`** — отклонено, так как это смешивает локальный путь и удалённый адрес; разделение явно выражает разные сущности.
- **Генерация ключей непосредственно в CI** — отклонено, так как это приводило бы к несовместимости подписей между релизами (каждый раз новый ключ). Правильный подход — однократная генерация и безопасное хранение.

## 5. Итог

Данный ADR фиксирует поведение утилиты `sign-tool`, обеспечивая единообразие работы как в CI/CD, так и в ручных сценариях. Решения по форматам ключей, обязательным параметрам, обработке существующих файлов и **правильному использованию ключей в CI** позволяют избежать неоднозначностей при разработке и эксплуатации.
